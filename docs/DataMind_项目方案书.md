# DataMind — 智能问数平台 项目方案书

> 期末项目 Proposal | 2026年5月

---

## 一、项目概述

### 1.1 项目名称

**DataMind — 智能问数平台**

### 1.2 一句话定位

一个基于 Flask 的 Web 应用程序，用户上传数据文件后，系统自动完成数据清洗与分析，用户通过**自然语言问答**与 AI 协作完成交互式数据探索，系统主动推送数据洞察并一键生成分析报告。

### 1.3 核心价值

传统数据分析工具（Excel、Tableau）要求用户掌握公式或拖拽操作。DataMind 的核心理念是：**让用户用"说人话"的方式分析数据**——不需要写代码、不需要学工具，只需提出问题，AI 自动将问题转化为 Pandas 代码并执行，返回结果和图表。

### 1.4 默认数据集

**Online Retail** — UCI 电商交易数据集（约 54 万条记录，8 个原始字段）。

选择理由：
- 数据量充足（远超 1000 行要求）
- 字段语义丰富（时间、客户、商品、国家、金额等多维度）
- 业务场景直观，适合做 RFM 客户分析、销售趋势、商品排行等分析
- Demo 演示时容易讲故事，老师和同学能直观理解分析结论

系统设计为**数据集无关**，支持用户上传任意 CSV/Excel/JSON 文件，自动适配数据结构。

## 二、需求分析与设计思考过程

### 2.1 项目要求拆解

项目要求实现"数据文件读取 → 数据预处理 → 数据分析方法 → 数据可视化 → 交互式分析"五大模块。我对每个模块进行了逐一思考：

| 模块 | 项目要求 | 我的思考 | 设计决策 |
|------|---------|---------|---------|
| 数据读取 | 能读取数据文件 | 只支持一种格式太局限，应支持多格式；编码问题是中文数据的常见痛点 | 支持 CSV/Excel/JSON，用 chardet 自动检测编码 |
| 数据预处理 | 完成数据处理 | 预处理步骤应可追溯，用户需要知道"数据被怎么处理了" | Pipeline 链式设计，每步记录日志，生成预处理摘要报告 |
| 数据分析 | 实现数据分析方法 | 内置常用分析方法（统计、趋势、RFM），但也要支持开放式分析 | 预置 7 种分析方法 + AI 动态生成任意分析 |
| 数据可视化 | 实现数据可视化 | 静态图表不够，需要交互式（悬停、缩放、导出） | 使用 Plotly.js，6 个预置仪表盘 + AI 在对话中动态生成图表 |
| 交互式分析 | 基础问答 | "基础问答"太简单了，应该做到"智能问答"——自然语言转代码 | GPT 将自然语言转 Pandas 代码，安全执行并返回结果 |

### 2.2 创新功能的思考

在五大核心模块之外，我设计了三个创新功能来争取 +10% 加分：

**创新 1：智能异常检测 + 自动洞察**

- **动机：** 传统 BI 工具是"被动分析"——用户问什么才看什么。DataMind 应该能"主动发现"——上传数据后系统自动扫描，主动推送有价值的发现。
- **实现思路：** 基于统计规则（IQR、Z-Score、环比阈值等）自动扫描所有数值列，发现异常值、趋势突变、分布集中等模式，生成洞察卡片。
- **关键决策：** 洞察生成不依赖 LLM API，纯规则+统计实现。理由是：速度快（毫秒级）、零成本、结果可解释可验证。

**创新 2：多轮对话记忆**

- **动机：** 单轮问答无法支持"追问"（如"去掉英国再看看"），这不符合真实的数据分析场景。
- **实现思路：** 每轮对话的问题 + AI 生成的代码 + 执行结果摘要都存入对话历史，构建完整上下文发送给 LLM。
- **关键决策：** 历史窗口限制为最近 20 轮，避免 token 超限。

**创新 3：自动生成分析报告**

- **动机：** 数据分析的最终交付物往往是一份报告，而不是零散的图表和数字。
- **实现思路：** 整合数据概览、预处理摘要、自动洞察、对话分析记录，由 GPT 组织成连贯叙述。
- **关键决策：** GPT 只负责"讲故事"（组织语言），数据真实性由代码保证。不让 GPT 自己编造数据。

### 2.3 技术栈选择的思考

| 决策点 | 备选方案 | 最终选择 | 理由 |
|-------|---------|---------|------|
| 后端框架 | Flask / FastAPI / Django | **Flask** | 轻量易懂，课程中接触过，单体部署简单，答辩时能解释每行代码 |
| 前端方案 | React / Vue / 原生 HTML+JS / Streamlit | **原生 HTML+JS** | 无需学习前端框架，代码量可控，Bootstrap 5 快速搭建 UI |
| 图表库 | Matplotlib / ECharts / Plotly | **Plotly.js** | 交互性最好（缩放/悬停/导出），Python 和 JS 生态都支持 |
| AI 服务 | OpenAI / Claude / DeepSeek / 本地规则引擎 | **OpenAI GPT** | 代码生成能力最强，生态最成熟，Python SDK 简单 |
| 整体架构 | 单体 / 前后端分离 / 微服务 | **单体 Flask** | 个人项目无需复杂架构，demo 稳定性最重要 |

### 2.4 架构方案对比

在确定架构时，我对比了三种方案：

**方案 A：单体 Flask 应用（最终选择）**
```
浏览器 ←→ Flask (API + 页面渲染) ←→ Pandas/数据层 ←→ OpenAI API
```
- 优点：一个 `python app.py` 启动全部，代码量 2000-3000 行 Python + 500 行 JS，debug 简单
- 缺点：前后端耦合（但个人项目无所谓）

**方案 B：前后端半分离**
```
浏览器 (独立 HTML/JS) ←→ Flask REST API ←→ Pandas ←→ OpenAI API
```
- 优点：职责清晰，更接近"真实工程"
- 缺点：需处理 CORS、调试链路变长，收益不大

**方案 C：微服务拆分**
```
浏览器 ←→ API Gateway ←→ 数据服务 / AI 服务 / 报告服务
```
- 优点：架构最"正规"
- 缺点：严重过度工程，3 周内完成风险极大

**选择理由：** 方案 A。评分标准看的是功能完整度、可视化效果和 AI 运用，不是架构复杂度。省下来的架构时间用于打磨三个创新功能。同时通过模块化目录结构（data/ ai/ routes/）保证代码清晰度。

### 3.3 设计原则

| 原则 | 在本项目中的体现 |
|------|----------------|
| **单一职责 (SRP)** | 每个 .py 文件只做一件事：loader 只管读取，preprocessor 只管清洗，互不耦合 |
| **依赖抽象 (DIP)** | AI 层通过统一接口调用 OpenAI，换模型只改 config.py，业务代码不变 |
| **KISS** | 单体应用，不引入容器/消息队列/数据库，CSV 即数据源 |
| **YAGNI** | 不做用户注册登录、不做权限管理、不做持久化数据库——只做评分要求的功能 |
| **DRY** | 数据分析结果缓存复用，图表渲染逻辑统一封装在 charts.js 中 |

## 四、五大核心模块设计

### 4.1 模块一：数据文件读取

**职责：** 接收用户上传的数据文件，自动识别格式和编码，返回统一的 DataFrame。

**设计要点：**
- 支持三种格式：CSV、Excel (.xlsx)、JSON
- 使用 chardet 库自动检测编码，解决中文乱码问题
- 编码检测失败时按 utf-8 → gbk → latin1 顺序尝试
- 对外暴露统一接口 `load_file(path) → DataFrame`，屏蔽格式差异
- 上传文件通过 Flask 接收，保存到临时目录后调用 loader

**为什么这样设计：** 用户可能上传各种格式的文件，系统不应要求用户转换格式。统一接口让上层模块不需要关心底层读取细节。

### 4.2 模块二：数据预处理

**职责：** 对原始数据进行清洗，产出可供分析的干净数据。

**设计要点：**

采用 **Pipeline 链式处理** 模式：

```
原始数据 → 去重 → 缺失值处理 → 类型转换 → 异常值标记 → 特征工程 → 清洁数据
```

| 步骤 | 通用逻辑（适配任意数据集） | Online Retail 示例 |
|------|-------------------------|-------------------|
| 去重 | 全行去重（默认） | 基于 InvoiceNo + StockCode |
| 缺失值 | 数值列填中位数，文本列填 "Unknown"，高缺失率列(>50%)警告 | CustomerID 缺失标记保留 |
| 类型转换 | 自动推断日期列；数值列 to_numeric | InvoiceDate → datetime |
| 异常值标记 | 数值列 IQR 法标记（不自动删除） | Quantity ≤ 0 过滤 |
| 特征工程 | 检测到日期列 → 生成 Year/Month/DayOfWeek/Hour | TotalAmount = Quantity × UnitPrice |

**关键设计决策：**
- **链式调用 API：** 每个处理步骤返回 self，支持 `preprocessor.remove_duplicates().handle_missing().run_all()` 写法
- **处理日志：** 每步记录处理了多少行，最终生成预处理摘要报告，让用户知道数据被怎么处理了
- **通用适配：** 初始化时自动检测列类型（数值/文本/日期），不硬编码列名。换数据集时无需修改代码

**为什么这样设计：** 链式调用直观清晰，每步可独立调试。处理日志增加可解释性——答辩时能清楚说明"我对数据做了什么处理"。

### 4.3 模块三：数据分析方法

**职责：** 提供 7 种内置分析方法，覆盖常见数据分析需求。

| 分析类型 | 实现方法 | 输出 |
|---------|---------|------|
| 描述性统计 | df.describe() + 自定义汇总 | 各列均值/中位数/标准差/分布 |
| 时间趋势 | 按月/周/日聚合数值列 | 时间序列数据 |
| 排行分析 | groupby + nlargest | Top N 排行 |
| RFM 客户分析 | Recency / Frequency / Monetary 三维打分 | 客户分群结果 |
| 分类分布 | groupby 分类列聚合 | 各类别占比 |
| 相关性分析 | 数值列 pearson 相关系数 | 相关性矩阵 |
| 时段模式 | 按 Hour/DayOfWeek 聚合 | 周期性规律 |

**为什么选择这 7 种：** 它们覆盖了数据分析的核心维度——描述、趋势、排名、分群、分布、关联、周期。对于电商数据尤其有针对性（RFM 是电商分析的经典模型）。

### 4.4 模块四：数据可视化

**职责：** 提供 6 个预置交互式图表仪表盘 + AI 动态生成图表。

**预置图表：**

| 图表 | 类型 | 展示数据 |
|------|------|---------|
| 月度销售趋势 | 折线图 | TotalAmount 按月聚合 |
| Top 10 商品 | 水平柱状图 | 销售额前 10 商品 |
| 国家/地区分布 | 饼图 | 各国销售额占比 |
| RFM 客户分群 | 散点图 | Frequency vs Monetary，颜色=Recency |
| 每周购买模式 | 热力图 | DayOfWeek × Hour 订单量 |
| 单价分布 | 直方图+箱线图 | UnitPrice 分布 |

**技术选型 — Plotly.js：**
- 交互性好：支持缩放、悬停查看数据、框选、导出图片
- Python 和 JS 生态统一：后端可用 plotly.py 生成 JSON 配置，前端用 Plotly.js 渲染
- 比 Matplotlib 更适合 Web（Matplotlib 生成静态图片，需要额外处理）

### 4.5 模块五：交互式分析（智能问答）

**职责：** 用户用自然语言提问，AI 将问题转化为 Pandas 代码，安全执行后返回结果和图表。

**这是整个项目的核心亮点。**

**工作流程：**

```
用户提问（自然语言）
    ↓
构建 Prompt（系统提示 + 数据结构信息 + 对话历史 + 当前问题）
    ↓
调用 OpenAI API → 返回 Python/Pandas 代码
    ↓
安全校验（禁止 import os / subprocess / open 等危险操作）
    ↓
在受限命名空间中 exec() 执行（只暴露 df、pd、np，超时 5 秒）
    ↓
捕获执行结果（DataFrame / 数值 / 文本）
    ↓
判断是否需要可视化 → 生成 Plotly 图表配置
    ↓
返回给前端（文字回答 + 可选图表 + 可查看的代码）
```

**安全机制设计：**
- **黑名单校验：** 禁止 `import os`、`subprocess`、`open(`、`__import__`、`eval(`、`exec(` 等危险操作
- **受限命名空间：** exec() 只暴露 `df`（数据）、`pd`（pandas）、`np`（numpy），无法访问文件系统或网络
- **超时保护：** 代码执行超过 5 秒自动终止，防止死循环

**System Prompt 设计思路：**
- 告诉 GPT 当前数据集的完整信息（列名、类型、行数、前 5 行示例数据）
- 约束 GPT 只使用 pandas 和 numpy
- 约束结果赋值给变量 `result`
- 约束代码简洁（一般不超过 10 行）

**为什么这样设计：** 自然语言转代码是 LLM 最擅长的能力之一。通过安全沙箱执行，既利用了 AI 的灵活性，又保证了系统安全性。

## 五、三大创新功能设计

### 5.1 创新一：智能异常检测 + 自动洞察

**核心理念：** 系统不应被动等待用户提问，而应主动发现数据中的有价值模式。

**三种检测方法：**

| 方法 | 适用场景 | 原理 |
|------|---------|------|
| IQR 法 | 数值型异常值 | Q1 - 1.5×IQR 到 Q3 + 1.5×IQR 之外判定为异常 |
| Z-Score | 极端值检测 | 标准化后绝对值 > 3 判定为异常 |
| 趋势突变 | 时间序列 | 滑动窗口均值，偏离超过 2 倍标准差标记为突变点 |

**五种洞察类型：**

| 类型 | 示例输出 | 生成方式 |
|------|---------|---------|
| 趋势洞察 | "2011年11月销售额环比增长47%" | 时间序列环比 + 阈值 |
| 异常洞察 | "商品 23843 单价异常偏高（$65，均值 $3.5）" | 异常检测器输出 |
| 分布洞察 | "82% 的订单来自英国，市场高度集中" | 占比计算 + 集中度阈值 |
| 相关洞察 | "订单数量与单价呈弱负相关（r=-0.12）" | 相关性矩阵 |
| 周期洞察 | "周四是一周中订单最多的一天" | 时段分析 |

**关键决策：** 洞察引擎完全基于**规则 + 统计阈值**，不依赖 LLM API。
- 速度快（毫秒级，用户上传数据后立即展示）
- 零额外成本（不消耗 API token）
- 结果可解释（每条洞察可追溯到具体的统计依据）

### 5.2 创新二：多轮对话记忆

**核心理念：** 真实的数据分析是一个连续的探索过程，不是孤立的一问一答。

**示例对话链：**

```
用户: "销售额最高的国家是哪个"
AI:   "英国，占比 82%"         ← 第 1 轮

用户: "那去掉英国呢"           ← 追问，AI 理解"那"指上一个问题
AI:   "去掉英国后，荷兰排第一"  ← 第 2 轮

用户: "画个柱状图"             ← AI 理解是对当前结果画图
AI:   [生成柱状图]             ← 第 3 轮
```

**实现机制：**
- 每轮对话存储三部分：用户问题 + AI 生成的代码 + 执行结果摘要
- 发送给 LLM 时带上完整上下文（system prompt + 最近 20 轮历史）
- 历史窗口超过 20 轮时，保留最近的，丢弃最早的（防止 token 超限）

### 5.3 创新三：自动生成分析报告

**核心理念：** 数据分析的最终交付物是报告，而非零散的图表和数字。

**报告结构：**
1. **数据概览** — 行数、列数、时间范围、数据质量评分
2. **预处理摘要** — 做了哪些处理、影响了多少行
3. **关键洞察** — 来自自动洞察引擎的发现（附带图表）
4. **对话分析记录** — 用户问了哪些问题、得出了什么结论
5. **总结与建议** — GPT 基于以上信息撰写的总结

**GPT 的角色边界：** GPT 只负责将结构化数据（统计数字、分析结论）组织成连贯的叙述文本。所有数据来源于代码计算，GPT 不被允许编造数据。这保证了报告的**数据准确性**。

## 九、错误处理与鲁棒性设计

| 场景 | 处理策略 | 用户体验 |
|------|---------|---------|
| 上传非法文件格式 | 前端校验后缀 + 后端二次校验 | 弹出提示"请上传 CSV/Excel/JSON 文件" |
| CSV 编码错误 | chardet 检测 → utf-8 → gbk → latin1 顺序尝试 | 自动处理，用户无感知 |
| AI 生成的代码执行出错 | 捕获异常，不暴露错误栈 | "抱歉，我无法处理这个问题，请换一种方式提问" |
| AI 生成危险代码 | 安全校验拦截，记录日志 | 同上，用户不知道具体原因 |
| OpenAI API 超时/限流 | 重试 1 次，仍失败则降级 | "AI 服务暂时不可用，请稍后再试" |
| 列名不匹配预设分析 | 动态检测可用列，不可用则跳过 | 只展示可执行的分析方法 |
| 空数据集 / 全缺失列 | 预处理阶段检测 | 弹出提示说明问题 |
| 用户上传超大文件 | 限制文件大小（50MB） | 弹出提示"文件过大" |

## 十、性能优化策略

| 问题 | 方案 | 原理 |
|------|------|------|
| 大文件读取慢 | DataFrame 缓存在 Flask 全局变量 | 单用户场景足够，避免重复读取 |
| 重复分析计算 | analyzer 结果缓存，数据不变不重算 | 字典缓存，以分析方法名为 key |
| AI 响应慢（2-5秒） | 前端显示打字动画 | 让用户感知"AI 在思考" |
| 图表渲染卡顿 | >10000 数据点时随机采样 | 保证浏览器流畅渲染 |

## 十五、AI 使用边界的思考

### AI 擅长做的（充分利用）
- 将自然语言问题转化为 Pandas 代码（这是 LLM 的强项）
- 生成报告的叙述性文本（语言组织能力强）
- 辅助编写代码模板和样板代码

### AI 不擅长做的（我来做）
- 系统架构设计（AI 擅长局部代码，不擅长全局规划）
- 安全机制设计（AI 不会主动考虑代码注入风险）
- 预处理策略选择（需要理解业务含义，不是简单的技术问题）
- 洞察阈值设定（需要数据分析经验判断什么是"异常"）
- 错误处理的用户体验（需要站在用户角度思考）

### 关键设计决策由我做出
1. 选择单体架构而非微服务 — 基于项目规模和答辩需求
2. 洞察引擎用规则而非 LLM — 基于速度、成本和可解释性的权衡
3. 代码执行的安全沙箱设计 — AI 不会主动考虑安全问题
4. 预处理通用适配机制 — 需要工程思维设计灵活的接口
5. GPT 只做"讲故事"不做"算数据" — 明确 AI 的角色边界
